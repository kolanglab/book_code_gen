# 実行時コンパイルと高速化

最後の章では、いちばん発展的な話題を扱います。これまでの最適化は、すべて
**プログラムを実行する前**に行うものでした。これに対し、**プログラムを実行している
最中に**コードを生成・改良する技術があります。その代表が **JIT コンパイル
（Just-In-Time compilation）**です [Aycock, 2003](#cite:aycock2003)。なぜそんな
ことをするのか、そしてどう実現するのかを見ていきましょう。

## まずインタプリタを速くする

JIT に進む前に、第1部で作ったような**バイトコードインタプリタ**そのものを速くする
工夫から始めます。第1部の VM は、命令を1つずつ取り出して `case` 文で振り分けて
いました。

```ruby
while pc < insns.size
  op, arg = insns[pc]
  case op
  when :push_int then ...
  when :add      then ...
  # ...
  end
  pc += 1
end
```

この方式は分かりやすいのですが、命令を実行するたびに「命令を取り出し、`case` で
種類を判定し、対応する処理へ飛ぶ」という**ディスパッチ**の手間がかかります。
ディスパッチとは、次に実行すべき処理へ制御を振り分けることです。実行する命令の
数が多いほど、この手間が積み重なります。

これを軽くする古典的な技法が**ダイレクトスレッディング（direct threading）**です。
おおまかには、命令の種類を表す番号の代わりに「その命令を処理するコードの場所
（アドレス）」を命令列に持たせ、ひとつの命令の処理が終わったら次の処理の場所へ
**直接**飛ぶ、という方式です。`case` による集中的な振り分けをなくせるため、
分岐予測（CPU が次の飛び先を先読みする仕組み）が当たりやすくなり、実行が速く
なります [Ertl and Gregg, 2003](#cite:ertl2003)。

> [!NOTE]
> ダイレクトスレッディングは、C のような計算済みラベルへ飛べる言語で実装されます。
> Ruby のような言語では素直には書けませんが、考え方——「集中ディスパッチを避け、
> 命令から命令へ直接つなぐ」——は普遍的です。Ruby の処理系 YARV も、こうした
> インタプリタ高速化の工夫を取り入れています [Sasada, 2005](#cite:sasada2005)。

もうひとつの定番が**スーパー命令（superinstruction）**です。`load_local` の直後に
`add` が来る、といった**頻出する命令の並び**を、ひとつの新しい命令にまとめて
しまうのです。ディスパッチ回数が減り、まとめた中で最適化もできます。これは
第8章ののぞき穴最適化を、バイトコードのレベルで行っているとも言えます。

## なぜ実行時にコンパイルするのか

インタプリタをいくら工夫しても、命令を1つずつ解釈する以上、その CPU 向けに直接
コンパイルされた機械語にはかないません。一方で、「最初から全部を機械語へコンパイル
する」方式（**事前コンパイル, Ahead-Of-Time, AOT**）にも弱点があります。コンパイルに
時間がかかり、起動が遅くなりますし、実行してみないと分からない情報を使った最適化が
できません。

JIT コンパイルは、この両者の「いいとこ取り」を狙います [Aycock, 2003](#cite:aycock2003)。
基本的なアイデアは次のとおりです。

1. プログラムは、まずインタプリタで実行を始める（起動が速い）。
2. 実行しながら、**どこが何回実行されたか**を数える（プロファイリング）。
3. 何度も実行される「**ホットスポット**」だけを、実行時に機械語へコンパイルする。
4. 以降、その部分はコンパイル済みの機械語で高速に実行する。

ほとんどのプログラムでは、実行時間の大半がごく一部のコード（ループや頻繁に呼ばれる
関数）に集中します。そこだけを狙ってコンパイルすれば、コンパイルのコストを
最小限に抑えつつ、効果の大きい部分を高速化できる、という戦略です。

```mermaid
graph LR
    A[バイトコード] --> B[インタプリタで実行]
    B --> C{ホットか?}
    C -->|まだ| B
    C -->|十分実行された| D[実行時に機械語へコンパイル]
    D --> E[機械語で高速実行]
```

## 実行時情報を活かす最適化

JIT が AOT に勝てる最大の理由は、**実行してみて初めて分かる情報**を最適化に使える
ことです。とくに動的型付き言語（Ruby や JavaScript など）では、これが効きます。

たとえば Ruby の `a + b` という式は、`a` と `b` の型によって意味が変わります。
整数どうしなら整数の足し算、文字列どうしなら連結です。事前には型が分からないので、
インタプリタは毎回「`a` の型は何か」を調べてから処理を選びます。

ところが実行時にプロファイルを取ると、「この `a + b` は、これまで毎回 整数どうし
だった」と分かることがよくあります。そこで JIT は、「**整数だと仮定した**高速な
機械語」を生成します。仮定が当たっている限り、型を調べる手間なしに高速に動きます。

もちろん仮定が外れることもあります。あるとき `a` が文字列になったら、その高速版は
使えません。そこで JIT は仮定が崩れたことを検知し、安全なインタプリタ実行へ
**引き返します**。これを**脱最適化（deoptimization）**と呼びます。「楽観的に仮定して
速いコードを作り、外れたら戻る」というこの仕組みが、動的言語の JIT の核心です。

> [!IMPORTANT]
> 脱最適化があるからこそ、JIT は「ほとんどの場合に成り立つが、絶対ではない仮定」に
> 基づく大胆な最適化ができます。仮定が外れても正しく動くという安全網があるおかげで、
> 速度を攻めた最適化が許されるのです。

## メソッド JIT とトレーシング JIT

「どの単位でコンパイルするか」によって、JIT には大きく2つの流儀があります。

**メソッド JIT** は、関数（メソッド）を単位としてコンパイルします。ホットな関数が
見つかったら、その関数全体を機械語にします。境界が分かりやすく、伝統的な最適化を
適用しやすいのが利点です。多くの Java VM や、Ruby のメソッド単位の JIT がこの流儀です。

**トレーシング JIT** は、関数ではなく**実際にたどった実行経路（トレース）**を単位と
します。ホットなループを実行しながら、そのとき通った命令の並びを一本道として記録し、
その一本道を機械語にコンパイルします。ループ内の `if` のうち、実際にはほとんど通る
側だけを直線的なコードにできるので、分岐の少ない高速なコードになりやすい一方、
途中で経路を外れたときの処理（やはり脱最適化）が必要です。

どちらが優れているという話ではなく、対象とする言語やプログラムの性質に応じて
選ばれます [Aycock, 2003](#cite:aycock2003)。

## 言語実装基盤という発展形

JIT は実装が大変です。言語ごとに、ターゲット CPU ごとに、高品質な機械語生成器を
作るのは膨大な労力がかかります。そこで近年は、「**JIT の難しい部分を共通基盤に
任せ、言語ごとの仕事を小さくする**」という方向の研究が進んでいます。

その代表が Truffle と Graal です [Würthinger et al., 2013](#cite:wuerthinger2013)。
おおまかには、言語の実装者は第1部で作ったような**素朴な AST インタプリタ**を書く
だけでよく、それを実行しながら共通基盤が自動的に最適化し、機械語の JIT コンパイルまで
行ってくれる、という仕組みです。本書の出発点だった「AST をたどって実行・生成する」
という素朴な考え方が、最先端の処理系の入り口にもなっているのは興味深いことです。

汎用のコンパイラ基盤としては、第7章でも触れた LLVM も重要です
[Lattner and Adve, 2004](#cite:lattner2004)。LLVM は AOT・JIT の両方で使え、多くの
言語処理系がそのバックエンドとして採用しています。
